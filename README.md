# AgentTrace · 使用说明

给 Agent 装一个体检中心。本文面向产品、运营与业务方，逐章解释三大菜单的定位、每个指标的计算口径，以及如何用本平台一步步定位 Agent 的真实问题。

## 01 平台速览

### 这个平台到底是干什么的？
如果把 Agent 看作一个员工，AgentTrace 就是它的「体检中心」：通过 OpenTelemetry 链路埋点，逐次会话评估它的响应、稳定、思考、资源、编排五项体征，并把异常归堆，方便业务方快速定位"这一周哪类问题在恶化"。

### 1.1 一句话定位
Agent 的运行链路很长，单看某一次回答好不好并不能反映系统真实质量。AgentTrace 把每一次完整对话当作一份"体检报告"，在你看不到的"幕后流程"里，量化它**用了多久**、**调了几个工具**、**错在哪一步**、**烧了多少 Token**。
当业务方提出"最近 Agent 是不是变笨了"、"为什么客户等了这么久"这类问题时，本平台给你**可量化、可追溯、可分组**的答案。

💡 **适合谁用**
- **产品 / 运营**：监控线上 Agent 体检趋势，发现劣化及时止血。
- **算法 / 研发**：定位单次会话的失败根因，输出工程改进项。
- **业务方**：用统一口径汇报 Agent 服务质量，对外承诺 SLA。

### 1.2 三大菜单一图流
本平台只有三个菜单，按"由远到近"排列，构成一条天然的下钻链路：

| 菜单 | 看什么 | 解决什么问题 | 典型行为 |
| --- | --- | --- | --- |
| **大盘** | 整体趋势 + 5 维健康度 + 高风险 session 排行 | "这周 Agent 状态怎么样？" | 发现劣化趋势 / 定位重点 session |
| **明细** | 单次会话的完整体检报告（DAG / 5 维 / 异常清单 / 质量评估） | "这个 session 到底错在哪一步？" | 逐 span 追因 / 找关键路径 |
| **异常** | 把命中相同异常标签的 session 归堆，看共性 | "哪一类问题正在变多？" | 共性诊断 / 推动批量修复 |

⚠️ **容易混淆**
"异常"不是 AI 自动聚类，而是按 `failed_label`（如「执行步数过多」「上下文超限」）做的简单归堆。它解决的不是"找新问题"，而是"已知问题谁在中招"。

## 02 核心概念

### 先把"刻度尺"立起来
这一章是后面所有指标的地基。读完之后，你会对"会话 / 轮次 / 步骤 / 步骤埋点"四级结构、5 维健康度的含义、健康度分档、关键路径形成稳定理解。建议必读。

### 2.1 会话三层结构
理解 Agent 的执行结构，只需要记住三个层级：

| 层级 | 业务话 | 技术话 |
| --- | --- | --- |
| **Session（会话）** | 用户和 Agent 的**一次完整对话** | 一次 trace · 含一个根 span |
| **Turn（对话轮次）** | 用户问一句、Agent 答一句，算**一轮** | 同一个 session 下的一组 user→assistant 消息对 |
| **Step（执行步数）** | Agent 内心活动一次：**想 + 调工具** 算一步 | 一次 LLM call 加它紧随的 tool calls |
| **Span（步骤埋点）** | 步骤里的**每一个具体动作** | OpenTelemetry 链路上的最小单元 |

💡 **用对话理解一遍**
用户："帮我订张明天去上海的票" → Agent 想了下，调 `flight.search`、`flight.book`、`calendar.add`。
- 这是 **1 个 Session**
- 包含 **1 个 Turn**（用户问了一次）
- 包含 **1 个 Step**（Agent 一次思考触发了 3 个工具）
- 但有 **4 个 Span**（1 LLM + 3 tool）

### 2.2 5 维健康度
把"Agent 体检"拆成五项独立指标，每一维满分 100，综合分取均值：

| 维度 | 它在测什么 | 不及格代表 | 核心信号 |
| --- | --- | --- | --- |
| **响应** | 用户等了多久 | Agent **慢**，用户感知差 | 端到端耗时 ≤ 60s 满分 |
| **稳定** | 调用最终成不成功 | Agent **没办成事**（最终结果导向） | 工具调用失败率、根 span 异常退出 |
| **思考** | 步数合不合理 | Agent **想得太少或太多** | 执行步数（黄金区 3-9 步） |
| **资源** | Token 用得省不省 | Agent **烧得多但产出少** | 单轮 Token ≤ 76k 满分 |
| **编排** | 调度模式合不合理 | Agent **来回踩同一个坑**（过程导向） | 连续重试次数、工具多样性 |

⚠️ **稳定 vs 编排：很容易搞混**
稳定看后果，编排看模式。同一个 session 可能"跌跌撞撞最终成功"——稳定 100、编排 50。这种 case 真实发生。

### 2.3 健康度分档
每个 session 的综合分会被映射到四档颜色，大盘 / 明细 / 异常三个菜单的颜色编码完全一致，便于跨菜单直觉记忆：
- 🟢 **≥ 85 健康**：体检通过。继续监控。
- 🟠 **70 - 84 一般**：轻度异常，关注趋势。
- 🟣 **50 - 69 风险**：需要排查，避免劣化。
- 🔴 **< 50 严重**：立即介入，影响线上。

### 2.4 关键路径
关键路径（Critical Path）= DAG 里**最长的串行链路**，决定了用户感知的总耗时。理解它最直观的方式是把 Agent 的执行流程想成一场接力赛：
- 多人并行跑 = 同时调多个工具，最终耗时 = 最慢那个
- 接力串行跑 = 一个工具调完才能调下一个，耗时 = 所有耗时之和
- 关键路径就是这条最长的接力链 —— **它一秒不优化，端到端就一秒不变快**

✅ **怎么读关键路径占比**
- 占比 **> 80%**：典型的**串行瓶颈**，Agent 在反复 thinking 或 retry，重点查模型/重试逻辑
- 占比 **50%-80%**：编排相对健康
- 占比 **< 50%**：可能存在**排队 / 并发浪费**，多个工具同时启动但未带来速度收益

## 03 菜单使用指南

### "我打开它能干什么"
三大菜单逐一拆解。每个菜单按统一模板讲清：它在哪、看什么、怎么用、典型场景。

### 3.1 大盘（Dashboard）
日常监控的第一站。
- **顶部 4 卡 · 核心指标**：E2E P95, E2E P50, 总成本, 累计 Token。
- **健康度分布堆叠条 + 5 维雷达**：四档健康度占比，雷达图反映哪一维拖后腿。
- **高风险 session 表**：按健康度升序排列（最差的在最上面），异常列用 `+n` 折叠展示。

💡 **为什么按健康度而不是按耗时？**
耗时高的不一定是系统真问题（用户提了个复杂任务自然慢）。健康度综合了 5 维信号，更接近"这一次 Agent 是不是真的失常了"。

### 3.2 明细（Sessions）
单次会话的"X 光片"。
- **列表页**：支持筛选和搜索。
- **详情页模块拆解**：
  - **基本信息**：session_id、用户、起始时间等。
  - **5 维雷达**：本次 session 各维得分。
  - **DAG 甘特图**：所有 span 按时间轴展开，关键路径高亮。
  - **执行分析**：实时日志风格的事件流。
  - **质量评估**：规则评估 + LLM-as-Judge + 人工校准。
  - **异常清单**：命中的 `failed_label` 明细。

### 3.3 异常（Clusters）
"同病相怜"视角。
- **顶部 3 卡**：异常类型数、受影响 Session、平均健康度。
- **聚类卡片字段**：异常 label、命中 session 数、平均健康度、趋势 sparkline、命中 Session ID。

## 04 指标口径手册

### 较真的人请看这里
本章是平台所有指标的权威口径。

### 4.1 耗时类
| 指标 | 含义 | 口径公式 | 数据来源 |
| --- | --- | --- | --- |
| E2E 耗时 | 用户从提问到收到完整答复的总时长 | root_span.end_time − root_span.start_time | OTel root span |
| E2E P95 | 95 分位耗时 | percentile(durations, 0.95) | 当前时间窗内全部 session |
| E2E P50 | 中位耗时 | percentile(durations, 0.50) | 同上 |
| 关键路径耗时 | DAG 最长串行链总和 | DFS(DAG) → max(serial chain duration) | span 父子关系 |
| 关键路径占比 | 关键路径 / 总耗时 | critical_path / e2e | 同上 |

### 4.2 成本类
| 指标 | 含义 | 口径公式 | 数据来源 |
| --- | --- | --- | --- |
| 累计 Token | 输入 + 输出 token 之和 | Σ(input_tokens + output_tokens) | 每个 LLM span 的 attributes |
| 总成本（$） | 按统一单价折算 | total_token × $1.5 / 1M | 同上 · 单价为占位估算 |

### 4.3 步数类
| 指标 | 含义 | 口径 | 判定 |
| --- | --- | --- | --- |
| 执行步数 | Agent 一次会话的思考循环数 | trace_count | — |
| 步数合理 黄金区 | 3-9 步 | 3 ≤ steps ≤ 9 | 合理 |
| 执行步数过多 | 步数 > 9，疑似进入循环 | steps > 9 | 异常 |
| 执行步数过少 | 步数 < 3，疑似过早结束 | steps < 3 | 异常 |

### 4.4 健康度子项打分
| 维度 | 满分条件 | 计分公式 |
| --- | --- | --- |
| **响应** | ≤ 60s | 100 × (1 − max(0, duration − 60s) / 60s) |
| **稳定** | 无失败、根 span 正常 | 基于 tool 失败率 + 根 span 退出状态 |
| **思考** | 6 步 | 距黄金区 3-9 越近得分越高 |
| **资源** | ≤ 76k token / 轮 | 100 × (1 − max(0, tokens − 76k) / 76k) |
| **编排** | 低 max_serial_run + 高工具多样性 | 惩罚连续相同工具调用 (max_serial_run) |

### 4.5 异常标签枚举
| failed_label | 触发条件 | 归属维度 |
| --- | --- | --- |
| **执行步数过多** | steps > 9 | 思考 |
| **执行步数过少** | steps < 3 | 思考 |
| **上下文超限** | 单次 LLM 调用 token 超出模型上下文窗口 | 资源 |
| **执行失败** | 根 span 异常退出 / 任务未完成 | 稳定 |
| **出现死循环** | 同一工具连续调用 max_serial_run ≥ 阈值 | 编排 |

## 05 问题排查路径

### "我看到异常了，下一步怎么办？"
本章把日常使用中最常遇到的 3 类排查场景，做成可复用的下钻路径。

### 路径 A · 大盘看到 P95 飙升 → 怎么定位到具体 session
1. **在大盘看到 P95 异常**：顶部 4 卡的 P95 数字明显高于历史水位。
2. **下拉到「高风险 session 表」**：按健康度升序排列，异常 chip 列就是分诊单。
3. **点击 session id 进入明细详情页**：重点看 **5 维雷达**，是不是响应维度独凹下去？是的话进入路径 C。

### 路径 B · 异常菜单某个标签飙红 → 怎么找共性
1. **在异常看到趋势 sparkline 上升**：说明该问题正在变多。
2. **点击命中 Session ID 抽查 2-3 个**：看是否共用同一个工具 / 同一类用户输入 / 同同一个 Agent 版本。
3. **在每个 session 的 DAG 里定位失败 span**：定位错误信息，共性出现后推动批量修复。

### 路径 C · 单 session 健康度 < 50 → 怎么用 5 维雷达 + DAG 找根因
1. **在 5 维雷达上找最低维**：哪一维独凹下去就先查哪一维。
2. **在 DAG 上定位关键路径**：如果占比 > 80%，说明串行瓶颈，重点查耗时最长的 span。
3. **读「执行分析」事件流**：按时间顺序看，红色事件就是断点。
4. **读「异常清单」+「质量评估」**：两者交叉验证，避免单一信号误判。

✅ **排查心法**
1. **先看分布，再看个体**：大盘 → 异常 → 明细，避免被极端 case 带偏。
2. **哪一维独凹下去先查哪一维**：5 维雷达就是分诊。
3. **关键路径优先**：DAG 上节点很多但只有橙色链决定总耗时。
4. **规则信号 ∩ LLM 信号**：单一来源容易误报，交叉验证更可靠。

### B · FAQ
- **Q1 · 为什么我看到的总成本和实际财务账单对不上？**
平台用 `$1.5 / 1M token` 简化估算，不区分 input / output 实际单价，仅用于横向对比。
- **Q2 · 为什么这个 session 没有 DAG？**
OTel 上报数据残缺，通常是埋点版本不一致或日志被截断。
- **Q3 · "执行步数"和"对话轮次"的区别？**
对话轮次 = 用户问了几次；执行步数 = Agent 自己思考了几次（调了多少次大模型决策）。
- **Q4 · 健康度 ≥85 是不是就万事大吉？**
不一定。建议同时看 5 维雷达形状，明显凹陷的维度无论综合分多高都值得关注。
- **Q5 · 异常菜单的趋势 sparkline 数据从哪来？**
当前为按 session 真实 `started_at_ms` 时间戳分桶的命中数。
