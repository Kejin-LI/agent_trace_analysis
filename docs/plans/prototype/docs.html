AgentTrace · 使用说明

给 Agent 装一个体检中心。本文面向产品、运营与业务方，逐章解释三大菜单的定位、每个指标的计算口径，以及如何用本平台一步步定位 Agent 的真实问题。

01平台速览

这个平台到底是干什么的？
如果把 Agent 看作一个员工，AgentTrace 就是它的「体检中心」：通过 OpenTelemetry 链路埋点，逐次会话评估它的响应、稳定、思考、资源、编排五项体征，并把异常归堆，方便业务方快速定位"这一周哪类问题在恶化"。
本章 · 预计阅读 2 分钟

1.1 一句话定位

Agent 的运行链路很长，单看某一次回答好不好并不能反映系统真实质量。AgentTrace 把每一次完整对话当作一份"体检报告"， 在你看不到的"幕后流程"里，量化它用了多久、调了几个工具、错在哪一步、烧了多少 Token。 当业务方提出"最近 Agent 是不是变笨了"、"为什么客户等了这么久"这类问题时，本平台给你可量化、可追溯、可分组的答案。

💡 适合谁用
  • 产品 / 运营：监控线上 Agent 体检趋势，发现劣化及时止血。
  • 算法 / 研发：定位单次会话的失败根因，输出工程改进项。
  • 业务方：用统一口径汇报 Agent 服务质量，对外承诺 SLA。

1.2 三大菜单一图流

本平台只有三个菜单，按"由远到近"排列，构成一条天然的下钻链路：

菜单看什么解决什么问题典型行为
大盘 整体趋势 + 5 维健康度 + 高风险 session 排行 "这周 Agent 状态怎么样？" 发现劣化趋势 / 定位重点 session
明细 单次会话的完整体检报告（DAG / 5 维 / 异常清单 / 质量评估） "这个 session 到底错在哪一步？" 逐 span 追因 / 找关键路径
异常 把命中相同异常标签的 session 归堆，看共性 "哪一类问题正在变多？" 共性诊断 / 推动批量修复
⚠️ 容易混淆
"异常"不是 AI 自动聚类，而是按 failed_label（如「执行步数过多」「上下文超限」）做的简单归堆。它解决的不是"找新问题"，而是"已知问题谁在中招"。

02核心概念

先把"刻度尺"立起来
这一章是后面所有指标的地基。读完之后，你会对"会话 / 轮次 / 步骤 / 步骤埋点"四级结构、5 维健康度的含义、健康度分档、关键路径形成稳定理解。建议必读。
本章 · 预计阅读 4 分钟

2.1 会话三层结构

理解 Agent 的执行结构，只需要记住三个层级：

层级业务话技术话
Session（会话）用户和 Agent 的一次完整对话一次 trace · 含一个根 span
Turn（对话轮次）用户问一句、Agent 答一句，算一轮同一个 session 下的一组 user→assistant 消息对
Step（执行步数）Agent 内心活动一次：想 + 调工具 算一步一次 LLM call 加它紧随的 tool calls
Span（步骤埋点）步骤里的每一个具体动作OpenTelemetry 链路上的最小单元
💡 用对话理解一遍
用户："帮我订张明天去上海的票" → Agent 想了下，调 flight.search、flight.book、calendar.add。
  • 这是 1 个 Session
  • 包含 1 个 Turn（用户问了一次）
  • 包含 1 个 Step（Agent 一次思考触发了 3 个工具）
  • 但有 4 个 Span（1 LLM + 3 tool）

2.2 5 维健康度

把"Agent 体检"拆成五项独立指标，每一维满分 100，综合分取均值：

维度它在测什么不及格代表核心信号
响应用户等了多久Agent 慢，用户感知差端到端耗时 ≤ 60s 满分
稳定调用最终成不成功Agent 没办成事（最终结果导向）工具调用失败率、根 span 异常退出
思考步数合不合理Agent 想得太少或太多执行步数（黄金区 3-9 步）
资源Token 用得省不省Agent 烧得多但产出少单轮 Token ≤ 76k 满分
编排调度模式合不合理Agent 来回踩同一个坑（过程导向）连续重试次数、工具多样性
⚠️ 稳定 vs 编排：很容易搞混
稳定看后果，编排看模式。 同一个 session 可能"跌跌撞撞最终成功"——稳定 100、编排 50。这种 case 真实发生： 参考演示数据 441ce4，连续 7 次 retry 同一工具但最终成功。

2.3 健康度分档

每个 session 的综合分会被映射到四档颜色，大盘 / 明细 / 异常三个菜单的颜色编码完全一致，便于跨菜单直觉记忆：

≥ 85 健康
体检通过。继续监控。
70 - 84 一般
轻度异常，关注趋势。
50 - 69 风险
需要排查，避免劣化。
< 50 严重
立即介入，影响线上。

2.4 关键路径

关键路径（Critical Path）= DAG 里最长的串行链路，决定了用户感知的总耗时。 理解它最直观的方式是把 Agent 的执行流程想成一场接力赛：

  • 多人并行跑 = 同时调多个工具，最终耗时 = 最慢那个
  • 接力串行跑 = 一个工具调完才能调下一个，耗时 = 所有耗时之和
  • 关键路径就是这条最长的接力链 —— 它一秒不优化，端到端就一秒不变快
✅ 怎么读关键路径占比
  • 占比 > 80%：典型的串行瓶颈，Agent 在反复 thinking 或 retry，重点查模型/重试逻辑
  • 占比 50%-80%：编排相对健康
  • 占比 < 50%：可能存在排队 / 并发浪费，多个工具同时启动但未带来速度收益

💡 为什么按健康度而不是按耗时？
耗时高的不一定是系统真问题（用户提了个复杂任务自然慢）。健康度综合了 5 维信号，更接近"这一次 Agent 是不是真的失常了"。

DAG 怎么读

  • 「⚡ 直接调用工具」标签：模型隐式决策，没有产出文本但直接发起了 tool call

B · FAQ

Q1 · 为什么我看到的总成本和实际财务账单对不上？
平台用 $1.5 / 1M token 简化估算，不区分 input / output 实际单价，仅用于横向对比 session 间的资源消耗，不是财务真账。
Q2 · 为什么这个 session 没有 DAG？
OTel 上报数据残缺（只有 root span 没有子 span），通常是埋点版本不一致或日志被截断。可以联系研发同学确认。
Q3 · "执行步数"和"对话轮次"的区别？
对话轮次 = 用户问了几次（一来一回算一次）；执行步数 = Agent 自己思考了几次（用户问 1 次但 Agent 可能内部思考 7 步）。
Q4 · 健康度 ≥85 是不是就万事大吉？
不一定。综合分会被某些维度拉高。建议同时看5 维雷达形状：理想形状是接近正五边形，明显凹陷的维度无论综合分多高都值得关注。
Q5 · 异常菜单的趋势 sparkline 数据从哪来？
当前为按 session 真实 started_at_ms 时间戳分桶的命中数。原型阶段数据样本较少（仅 6 条），接入真实 API 后会自动铺满时间窗。
